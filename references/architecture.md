# 架构设计

## 登录状态复用机制

### 核心架构

```
login.sh (state save) → 登录状态文件 → approve.sh (state load)
                                        ↓
                                   独立session执行审批
```

### 工作原理

#### State Save/Load 机制

```bash
# login.sh: 保存登录状态
$AGENT_BROWSER --session "$SESSION_NAME" state save "$STATE_FILE"

# approve.sh: 加载登录状态
$AGENT_BROWSER --session "$SESSION_NAME" state load "$STATE_FILE"
```

**保存的内容**：
- Cookies
- LocalStorage
- SessionStorage
- 浏览器上下文状态

**有效期**：约30分钟（取决于OA系统session超时配置）

### 优势

1. **登录一次，多次使用**
   - 减少登录次数，降低账号风险
   - 提升执行效率

2. **独立session，互不干扰**
   - 每个审批任务使用独立session
   - 支持并发执行多个审批任务

3. **性能提升**
   - 跳过OA登录页面（节省约5-8秒）
   - 直接访问费控系统（节省约2-3秒）
   - 单个审批任务从 ~15秒 降至 ~7秒

## 并发架构

### Session 隔离机制

每个审批脚本创建独立的session，使用纳秒级时间戳确保唯一性：

```bash
SESSION_NAME="oa-approve-$(date +%s%N)"  # 例如: oa-approve-1739512345678901234
```

**优势**：
- 多个审批任务可同时运行，互不干扰
- 避免session冲突导致的操作混乱
- 每个任务有独立的日志和截图

### 并发控制建议

```bash
# 建议并发数不超过5
./scripts/batch_approve.sh approval_list.csv 3

# 大批量任务分批执行
# 每批20-30个，避免系统过载
```

## 核心流程对比

### query_approval.sh - 查询专用脚本

```
登录OA → 点击费控 → 新标签页 → 待办菜单 → 获取待审批列表
```

**用途**：
- ✅ 查询费控系统待审批流程
- ✅ 获取待审批单据列表
- ❌ 不执行审批操作

**特点**：
- 完整的登录到查询流程
- 适用于首次查询或状态过期时
- 输出待审批列表截图

### approve.sh - 审批专用脚本

```
加载状态 → 直接打开费控 → 待办 → 搜索单号
    ↓
找到单据？
    ├─ 是 → 勾选 → 执行审批 → 验证结果
    └─ 否 → 切换审批记录 → 搜索
              ↓
          找到单据？
              ├─ 是 → 显示已审批状态
              └─ 否 → 提示单据不存在
```

**用途**：
- ✅ 执行单据审批（同意/驳回）
- ✅ 查询单据审批状态
- ❌ 不用于获取待审批列表

**特点**：
- 使用已保存的登录状态（更快）
- 支持智能查询（待审批+审批记录）
- 适用于已知单号的审批操作

### 智能查询功能

当在待审批列表中未找到单据时，脚本会自动：
1. 切换到"审批记录"标签页
2. 在审批记录中搜索单号（默认提供半年内记录）
3. 如果找到单据，显示其审批状态（已同意/已驳回）
4. 如果未找到，提示可能的原因

**优势**：
- 快速区分"已审批"和"不存在"两种情况
- 审批记录默认提供半年内数据，无需额外操作
- 避免重复审批和无效操作

**性能影响**：
- 智能查询增加约2-3秒（仅在待审批中未找到时）
- 大部分情况下在待审批中找到，不影响性能

## 性能数据

### 时间对比

| 操作 | 传统方式 | 优化后 | 提升 |
|------|---------|--------|------|
| 单个审批 | ~15秒 | ~7秒 | 53% |
| 登录流程 | 每次登录 | 复用状态 | 100% |
| 批量审批（10个） | ~150秒 | ~35秒 | 77% |

### 资源使用

- **内存**：每个session约50-100MB
- **CPU**：中等负载，支持并发
- **网络**：每个审批任务约10-20次请求

## 扩展性

### 支持的并发数

- **推荐**：3-5个并发任务
- **最大**：取决于系统资源
- **建议**：大批量任务分批执行
