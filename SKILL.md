### 核心工作模式

**展示待办时**，你的目标是让用户快速了解当前待办状态：
- 获取待办列表和统计概览（显示总数、按类型分组）
- 用友好的表格展示，将技术术语翻译成用户容易理解的中文
- 提供清晰的后续操作提示（比如"输入序号查看详情"）

**处理用户的自然语言指令**时，理解用户的真实意图：
- 用户说"通过前5个"——批量操作前5条待办
- 用户说"通过张三提交的"——按提交人筛选后批量操作
- 用户说"第1、3、5个"——按序号批量操作

### 数据同步策略

同步待办有两种方式，各有适用场景：

**快速同步**（仅获取列表）：速度快，适合频繁刷新。当用户只想快速看看有什么待办时使用。

**完整同步**（获取详情）：速度较慢，适合首次同步或定时任务。详情包含表单内容、审批历史等，约200条约需几分钟。建议在非工作时间进行完整同步，工作时间使用快速同步保持数据新鲜。

### 交互式信息提取

当用户需要深入了解某个待办时——比如"看看议程"、"有什么附件"、"审批历史是什么"——你可以创建一个浏览器会话来交互式地提取信息。这比静态文本能提供更丰富的上下文，比如：
- 会议邀请：完整的议程安排、参会人员名单、会议材料下载
- 流程审批：详细的表单字段、完整的审批链路、附件预览
- 费用报销：逐项报销明细、票据影像

创建会话后，你可以浏览页面、提取关键信息，然后以结构化方式呈现给用户，帮助ta做出决策。

### OA页面探索

用户可能需要在OA系统中执行不属于待办审批的操作，比如"预约会议室"、"提交请假申请"、"填写报销单"。这种情况下，你可以创建一个探索会话，与用户进行多轮对话来理解ta的具体需求，然后生成相应的操作步骤让用户确认。

### 审批安全原则

审批操作具有不可逆性，因此**所有审批都需要用户明确确认**：

**单个审批**：先展示待办详情，等用户确认后再执行

**批量审批**：采用"预览-编辑-确认"流程
1. 展示待办列表和拟执行的动作
2. 允许用户调整特定待办的审批动作
3. 等用户最终确认后，使用 `--force` 逐条执行

这样设计的原因是：批量审批涉及多个待办，用户需要在执行前看到完整列表，并有机会修改自己的决策。二次确认虽然看似重复，但能有效避免误操作。

对于会议邀请，理解用户的自然表达：
- 用户说"同意"会议邀请 = 参加
- 用户说"不同意"会议邀请 = 不参加

## 快速开始

### 本地源码编译安装（推荐）

```bash
cd oa-todo
npm install
npm run build
npm link

# 配置环境变量 (在 CoPaw Environments 中)
# OA_USER_NAME=your_username
# OA_USER_PASSWD=your_password

oa-todo sync
```

### 配置定时任务（推荐）

安装完成后，建议配置定时任务自动同步待办，保持数据最新。

**使用 CoPaw Cron（推荐）**

CoPaw Cron 是 CoPaw 内置的定时任务系统，与 Agent 深度集成，支持任务管理、日志监控和状态追踪。

**配置方式**：

使用 `cron` 技能来配置定时任务，技能会自动处理参数配置和验证。

你可以这样说：
- "帮我配置 OA 定时同步"
- "创建一个每30分钟同步待办的任务"
- "设置工作日定时同步 OA 待办"

**同步时间参考**：

| 场景 | 建议频率 | 说明 |
|------|---------|------|
| 工作时间快速刷新 | 每15-30分钟 | 仅同步列表，快速获取待办 |
| 工作时间详情同步 | 每1-2小时 | 同步列表+详情，平衡性能 |
| 非工作时间全量 | 每天凌晨1次 | 完整同步所有待办详情 |

**⚠️ 注意**：确保环境变量（OA_USER_NAME、OA_USER_PASSWD）已在 CoPaw Environments 中配置。

## 🎯 典型工作流程

### 标准审批流程

当用户需要审批待办时，按照以下步骤进行：

1. **列出待审核事项**——使用 `oa-todo list` 获取待办列表，展示给用户
   - 参考命令：[commands.md#list](references/commands.md#list)

2. **用户选择某个待办**——使用 `oa-todo show <fdId>` 显示该待办的详细信息
   - 参考命令：[commands.md#show](references/commands.md#show)

3. **用户确认审批意见**——询问用户审批动作（通过/驳回等）
   - 根据待办类型确定可用动作（见 commands.md）

4. **执行审批**——使用 `oa-todo approve <fdId> <action>` 提交审批并反馈结果
   - 参考命令：[commands.md#approve](references/commands.md#approve)
   - **注意：**不要添加文档中未说明的参数选项

### 交互式批量审批

当用户需要批量审批时，采用"预览-编辑-确认"流程，让用户充分掌控审批决策：

**解析用户意图**
首先理解用户的自然语言审批意图，并筛选出相关的待办：
- "通过前5个" → 获取前5条待办，动作设为"通过"
- "驳回张三的" → 筛选提交人为张三的待办，动作设为"驳回"
- "参加会议" → 筛选类型为会议的待办，动作设为"参加"

**展示审批列表**
用表格清晰展示每个待办的信息和拟执行的动作，让用户一目了然：
```
序号 | 标题                    | 类型    | 拟执行动作
----|-------------------------|---------|------------
1   | 请审批服务器采购申请     | 流程审批 | 通过
2   | 邀请您参加会议           | 会议邀请 | 参加
3   | 张三的年假申请           | EHR假期 | 同意
```

**允许用户调整**
用户可以针对单个或多个待办修改审批动作：
- "把第2个改成不参加"
- "第3个先跳过，暂时不处理"
- "全部改成驳回，我要重新考虑"

**确认后执行**
用户最终确认后，逐条执行审批命令并报告结果。每条审批使用 `--force` 参数跳过二次确认，因为用户已经在预览阶段完成了确认。

## 核心设计原则

### 优先使用本地数据

待办数据会持久化在本地，避免每次都从OA系统获取。默认情况下，如果本地已有数据，同步命令会跳过已获取的待办。这既提高了响应速度，也减少了对OA系统的压力。只有当用户明确要求或数据过期时，才需要从服务器重新获取。

### 完整信息展示

用户需要足够的信息来做出审批决策。展示待办时应该包含：
- 完整的ID（方便引用和复制）
- 完整的标题（不截断，自动换行）
- 提交人信息和来源部门（帮助判断审批权限）

### 详细审批上下文

执行审批前，向用户展示完整的待办详情，包括提交人、来源部门、创建时间等，确保用户在充分了解情况的基础上做出决策。

### 常见错误模式（避免）

以下是一些常见的错误操作模式，请避免：

| 错误行为 | 正确做法 |
|---------|---------|
| 直接执行 `oa-todo approve <fdId> <action> --some-option` | 先查阅 commands.md 确认 `--some-option` 是否存在 |
| 使用 `oa-todo list --by-date`（假设有此选项） | 使用文档中实际存在的选项，如 `--sort-received` |
| 批量审批时直接循环执行 `approve --force` | 先执行"预览-编辑-确认"流程 |
| 推断"应该有某个功能"并尝试使用 | 先在 commands.md 中确认该功能是否存在 |
| 自行组合多个选项（如 `--a --b --c`） | 只使用文档示例中明确展示的组合 |

**记住：不要假设功能存在。如果文档没说，就当它不存在。**
