## ⚠️ 重要说明

### 智能体使用规范

**获取待办列表行为**：
1. 用户要求查看待办时，直接调用：`oa-todo list` 和 `oa-todo status`
2. 智能体自行构建优化表格（使用Markdown格式）
3. 翻译类型和状态为中文自然语言
4. 显示下一步操作提示

**类型翻译对照**：
- `workflow` → 流程审批
- `meeting` → 会议邀请
- `ehr` → EHR假期
- `expense` → 费用报销
- `unknown` → 未知类型

**状态翻译对照**：
- `pending` → 待审核
- `approved` → 已同意
- `rejected` → 已驳回
- `attended` → 已参加
- `not_attended` → 不参加
- `skip` → 已跳过
- `transferred` → 已转办
- `other` → 其他

**智能体内部处理**：
- 保存ID与序号的对应关系
- 用户通过序号操作时，自动映射到完整ID
- 使用Markdown表格格式输出，合理设置列宽

### 同步操作超时限制与定时同步策略

- **同步待办列表很快**，通常不会超时
- **获取待办详情较慢**：200条约需3分钟，可能超时
- **建议**：使用 `-c 5` 并发数，或 `--limit 25` 限制数量
- **推荐定时同步策略**：
  - 🌙 凌晨全量同步：`oa-todo sync --fetch-detail -c 5`（凌晨2点）
  - ☀️ 工作时间增量：`oa-todo sync --fetch-detail -c 5 --limit 25`（每小时，8:00-19:00）
  - 📋 仅同步列表：`oa-todo sync`（速度快，随时可执行）

### 复杂审批模式智能提示

**检测逻辑**：智能体在展示待办列表后，自动分析待办标题和类型，检测可能需要深度审查的待办。

**复杂度判断标准**：
- 标题包含：附件、采购、合同、技术方案、评审、服务器、设备、付款、报销等关键词
- 类型为：`workflow`（流程审批）或 `expense`（费用报销）
- 标题包含金额信息（如：¥10,000+）

**提示格式**：
```
🔍 检测到以下待办可能需要深度审查：

• [序号] 待办标题

💡 建议使用"复杂审批模式"进行深度分析

使用方法：复杂审批第N个
```

### 交互式信息提取（暂停模式）

**触发场景**：用户请求查看更多信息时，使用暂停模式创建浏览器会话进行交互式提取。

**用户意图触发词**："查看详情"、"深入了解一下"、"看看议程"、"有什么附件"

**工作流程**：
1. 创建暂停点会话：`oa-todo approve <fdId> --pause`
2. 使用 agent-browser 审查资料（提取页面信息、查看附件）
3. 向用户展示提取的信息
4. 用户决策后执行审批：`oa-todo approve <fdId> <action> --force`

**信息提取能力**：
- 会议邀请：议程、参会人员、会议材料
- 流程审批：表单详情、审批历史、附件列表
- 费用报销：报销明细、票据信息

### 通用页面探索模式

**触发场景**：用户需要操作OA系统中的通用页面（非待办审批）。

**用户意图触发词**："探索xxx页面"、"会议室预约"、"请假申请"、"费用报销"

**工作流程**：
1. 创建探索会话：`oa-todo explore <url> --pause`
2. 智能体解析 agentUXContext
3. 选择操作方式：
   - **HTTP API**（推荐用于数据查询）
   - **浏览器操作**（用于页面交互）
4. 使用 agent-browser 探索页面
5. 多轮对话理解用户意图
6. 生成命令序列并确认执行

**支持的场景**：会议室查询/预约、请假申请、费用报销、通用表单

### 审批安全原则

**审批操作具有不可逆性，是最高风险操作类型。**

**强制要求**：
1. 所有审批操作必须遵循"展示-确认-执行"三步流程
2. 确认流程的具体步骤和要求，请参见下方的「审批确认流程（强制执行）」章节
3. 禁止跳过确认直接执行审批

**单个审批 - 单个确认**：
1. 展示待办详细信息
2. 询问确认
3. 执行：`oa-todo approve <id> <action> --force`

**批量审批 - 一次确认**：
1. 展示所有待办的Markdown表格
2. 询问确认
3. 自动逐个执行

**批量审批支持自然语言**：
- 按序号：`"通过第1、3、5个"`
- 按范围：`"批量审批1-3号"`
- 按条件：`"通过所有张三提交的"`
- 按数量：`"批量审批前5个"`

## 🛡️ 审批确认流程（强制执行）

⚠️ **高风险操作警告**：审批操作具有不可逆性，可能导致严重的业务后果。

**核心原则**：所有审批操作（单个/批量）必须严格遵循"展示-确认-执行"三步流程，禁止跳过任何步骤。

---

### 🔒 强制性确认规则

**规则1：禁止直接执行审批命令**

在执行任何 `oa-todo approve` 命令前，必须先完成以下步骤：
1. 向用户展示待办的完整信息
2. 获得用户的明确确认
3. 使用 `--force` 参数执行（因为已完成确认）

**禁止行为**：
- ❌ 不得在未展示待办信息的情况下直接执行审批
- ❌ 不得根据用户自然语言推测意图后直接执行
- ❌ 不得省略"展示-确认-执行"中的任何步骤
- ❌ 不得使用模糊的确认语言（如"好的"、"收到"代替明确确认）

---

### 📋 单个审批确认流程（强制）

#### 步骤1：获取待办信息
```bash
oa-todo show <fdId>
```

#### 步骤2：格式化展示（必须包含以下所有字段）
```
📋 待办详细信息:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  序号:       [用户指定的序号]
  标题:       [完整标题，不截断]
  类型:       [流程审批/会议邀请/EHR假期/费用报销]
  当前状态:   [待审核]
  提交人:     [提交人姓名]
  来源部门:   [部门名称]
  创建时间:   [YYYY-MM-DD HH:MM:SS]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 步骤3：明确确认（必须使用以下格式）
```
确认要执行以下操作吗？
  操作: [通过/驳回/参加/不参加/同意/不同意]
  待办: [标题]

请回复 "确认" 继续，或输入其他指令修改。
```

#### 步骤4：执行审批（仅在用户明确确认后）
```bash
oa-todo approve <fdId> <action> --force
```

**重要**：只有用户明确回复"确认"、"同意"、"执行"、"继续"等肯定性词汇后，才能执行。用户说"好的"、"收到"等不构成审批确认。

---

### 📊 批量审批确认流程（强制）

#### 步骤1：获取待办列表
```bash
oa-todo list [根据用户意图筛选]
```

#### 步骤2：格式化展示表格
```
📋 批量审批待办列表 (共N个):
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

| 序号 | 标题 | 提交人 | 类型 | 状态 |
|------|------|--------|------|------|
| 1    | ...  | ...    | ...  | ...  |
...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
拟执行操作: [通过/驳回/参加/不参加/同意/不同意]
```

#### 步骤3：明确确认
```
确认批量执行以下操作吗？
  操作: [通过/驳回/参加/不参加/同意/不同意]
  数量: [N]个待办

请回复 "确认" 继续，或：
- 输入 "跳过X" 排除第X个
- 输入 "修改X 动作" 修改第X个的动作
```

#### 步骤4：逐个执行（仅在用户明确确认后）
```bash
oa-todo approve <id1> <action> --force
oa-todo approve <id2> <action> --force
...
```

### 🚨 常见错误模式（严格禁止）

| 错误行为 | 正确做法 |
|---------|---------|
| 用户说"通过第一个"，AI直接执行 | 先展示第一个的详细信息，等待用户确认 |
| 用户说"批量通过前5个"，AI直接执行 | 先展示5个待办的表格，等待用户确认 |
| 用户说"好的"，AI认为已确认 | "好的"不是审批确认，必须明确说"确认"/"执行" |
| AI用简化格式展示待办 | 必须使用上述标准化格式展示所有字段 |
| AI省略"请回复确认继续" | 必须明确要求用户回复确认 |

### 🔍 确认响应判定规则

**有效确认词汇**：
- "确认"
- "同意"
- "执行"
- "继续"
- "通过"（在操作为通过时）
- "可以执行"

**无效响应（需要再次确认）**：
- "好的" ❌
- "收到" ❌
- "OK" ❌
- "嗯" ❌
- "知道了" ❌

---

### 📝 审批确认检查清单

执行任何审批操作前，AI必须自检以下所有项目：
- [ ] 已获取待办的完整信息（单个用show，批量用list）
- [ ] 已使用标准化格式展示所有必需字段
- [ ] 已明确说明拟执行的操作类型
- [ ] 已明确要求用户回复"确认"
- [ ] 用户已明确回复有效确认词汇
- [ ] 使用 `--force` 参数执行

**任何一项未完成，禁止执行审批命令。**

### 会议邀请智能审批语义

**会议邀请的特殊审批映射**：
- "同意" → 执行 `参加`
- "不同意" → 执行 `不参加`
- "参加" → 执行 `参加`
- "不参加" → 执行 `不参加`

## 🎯 最佳实践工作流程

### 标准审批流程（推荐）

```bash
# 步骤1: 列出待审核
oa-todo list --type workflow --status pending

# 步骤2: 查看详情
oa-todo show <完整fdId>

# 步骤3: 用户确认后执行
oa-todo approve <完整ID> 通过
```

### 批量审批流程

```bash
# 步骤1: 列出所有待审核
oa-todo list --status pending

# 步骤2: 通过Markdown表格展示并确认
# 智能体展示列表 → 用户确认 → 智能体逐个执行

# 步骤3: 用户确认后自动执行
# oa-todo approve <id1> 通过 --force
# oa-todo approve <id2> 通过 --force
```

## 核心优化

### ✅ 优先使用本地数据
- 默认行为：本地有数据时，`oa-todo sync` 不自动同步
- 强制同步：`oa-todo sync --force`

### ✅ 完整信息展示
- 完整ID：列表显示32位ID，方便复制
- 完整标题：自动换行，不截断
- 提交人信息：新增提交人/来源部门列

### ✅ 详细审批信息
- 审批时展示完整详情
- 包含提交人、来源部门、创建时间
