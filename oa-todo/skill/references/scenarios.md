# 智能体同时执行两个命令：

oa-todo status

# 智能体获取CLI数据后，展示统计信息和优化表格：

# 1. 统计信息
📊 待办统计信息：
- 待审核总数：15个
- 流程审批：12个
- 会议邀请：2个
- 费用报销：1个

# 2. Markdown表格（隐藏ID，新增序号，保留核心字段）
| 序号 | 标题                                      | 提交人 | 接收时间         | 类型     | 状态   |
|------|------------------------------------------|--------|------------------|----------|--------|
| 1    | 请审批[基础架构部]吴庆提交的流程         | 吴庆   | 2025-01-15 10:30 | 流程审批 | 待审核 |
| 2    | 请审批[产品部]张三提交的流程             | 张三   | 2025-01-15 09:20 | 流程审批 | 待审核 |

下一步操作：
- 查看详情：查看第1个

- 批量审批：
  · "通过第1、3、5个"

# - 类型和状态翻译为中文

## 场景2：查看并审批单个待办（带强制确认）

```bash
# 1. 查看待办列表（同时查询list和status）
oa-todo list
oa-todo status

# 用户看到待办列表后，使用自然语言：
# 用户：查看第1个

# 2. 智能体执行查看详细信息
oa-todo show 18798bf8

# 3. 智能体标准化展示待办信息并请求确认
# 智能体输出：
#
# 📋 待办详细信息:
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
#   序号:       1
#   标题:       请审批[基础架构部]吴庆提交的流程：评审小组
#   类型:       流程审批
#   当前状态:   待审核
#   提交人:     吴庆
#   来源部门:   基础架构部
#   创建时间:   2025-01-15 10:30:00
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
#
# 确认要执行以下操作吗？
#   操作: 通过
#   待办: 请审批[基础架构部]吴庆提交的流程：评审小组
#
# 请回复 "确认" 继续，或输入其他指令修改。

# 4. 用户确认审批
# 用户：确认

# 5. 智能体执行审批（用户已明确确认，使用--force参数）
oa-todo approve 18798bf8c8b26e8fbfb50d34f0888f8f 通过 --force

# 说明：
# - 单个审批必须先展示详细信息，再请求用户确认
# - 必须使用标准化格式展示所有字段
# - 必须明确要求用户回复"确认"
# - 用户明确确认后使用--force参数执行
```

## 场景3：批量审批（Markdown表格展示+一次确认）

```bash
# 1. 查看待办列表（同时查询list和status）
oa-todo list
oa-todo status

# 2. 用户看到待办列表后，使用自然语言请求批量审批
# 用户示例1："通过第1、3、5个"
# 用户示例2："批量审批1-3号"
# 用户示例3："通过所有张三提交的"
# 用户示例4："批量审批前5个"

# 3. 智能体解析自然语言并通过Markdown表格展示待办列表
# 智能体：我将批量通过以下3个待办：

| 序号 | 标题                        | 提交人 | 类型     | 状态   |
|------|----------------------------|--------|----------|--------|
| 1    | 请审批[基础架构部]吴庆...  | 吴庆   | 流程审批 | 待审核 |
| 2    | 请审批[产品部]张三...      | 张三   | 流程审批 | 待审核 |
| 3    | 会议邀请：周五例会          | 王五   | 会议邀请 | 待审核 |

确认批量通过这3个待办吗？(y/n)

# 4. 用户输入 y 确认后，智能体自动逐个执行（使用--force参数）
oa-todo approve id1 通过 --force
oa-todo approve id2 通过 --force
oa-todo approve id3 通过 --force

# 5. （可选）智能体也可带审批意见逐个执行
oa-todo approve id1 通过 --comment "同意" --force
oa-todo approve id2 通过 --comment "同意" --force
oa-todo approve id3 通过 --comment "同意" --force

# 说明：
# - 批量审批时，智能体通过Markdown表格展示待办列表
# - 用户只需确认一次
# - 智能体使用--force参数自动逐个执行审批
```

## 场景4：处理会议邀请

```bash
# 1. 查看会议邀请（同时查询list和status）
oa-todo list --type meeting
oa-todo status

# 用户看到会议列表后，使用自然语言：
# 用户：查看第1个会议详情

# 2. 智能体执行查看会议详情
oa-todo show <会议ID>

# 3. 用户确认参加（支持多种自然语言表达）
# 用户示例1：参加第一个会议
# 用户示例2：同意参加第一个会议
# 用户示例3：同意第一个会议邀请
# 用户示例4：不同意这个会议

# 智能体将用户的自然语言映射到对应的审批动作：
# - "同意" → "参加"
# - "不同意" → "不参加"
# - "参加" → "参加"
# - "不参加" → "不参加"

# 智能体执行确认参加（用户已确认，使用--force参数直接执行）
oa-todo approve <会议ID> 参加 --force
```

## 场景5：配置定时自动同步（推荐）

**⚠️ 时区说明**：以下 cron 配置的时间均为**北京时间（UTC+8）**。

```bash
# 使用 cron 技能设置定时同步，避免手动同步超时

# 步骤1：编辑crontab
crontab -e

# 步骤2：添加以下定时任务

# 策略1：每天凌晨2点（北京时间）全量同步详情
0 2 * * * oa-todo sync --fetch-detail -c 5

# 策略2：工作时间每小时同步列表+25条详情（周一到周五，北京时间 8:00-19:00）
0 8-19 * * 1-5 oa-todo sync && oa-todo sync --fetch-detail -c 5 --limit 25

# 步骤3：验证配置
crontab -l  # 查看当前任务
tail -f /tmp/oa-sync.log  # 查看同步日志（如果配置了日志输出）

# 说明：
# - 凌晨全量同步：确保每天有完整的待办详情数据
# - 工作时间增量同步：保持待办列表最新 + 获取最新25条详情
# - 避免工作时间全量同步导致超时
# - -c 5 使用5个并发，平衡速度和稳定性
```

## 场景6：复杂审批流程（人工+智能体交互）

**适用场景**：需要深度审查的复杂审批，包含大量附件、文档、技术资料的审批。

**示例**：服务器采购、合同审批、技术方案评审等。

详细的复杂审批流程请参考：**[references/complex-approval.md](complex-approval.md)**

### 快速示例

```bash
# 用户: 复杂审批第1个待办

# 智能体: 我看到这是一个服务器采购申请，包含采购合同、技术规格说明书等文档，
# 智能体: 建议使用复杂审批流程进行深度审查。您需要吗？

# 用户: 好的

# 步骤1: 创建断点
oa-todo approve 18798bf8c8b26e8fbfb50d34f0888f8f --pause --timeout 15

# 步骤2: 智能体获取页面内容并总结
# 步骤3: 用户指定关注点
# 步骤4: 智能体进行深度分析
# 步骤5: 用户确认审批决策
# 步骤6: 执行审批并关闭断点
```

**关键特性**：
- 智能体主动提取和总结审批资料
- 交互式审阅：用户指定关注点，智能体深度分析
- 智能建议：基于分析结果给出审批建议
- 支持暂停审批并保存断点

## 环境变量

| 变量 | 必需 | 默认值 | 说明 |
|------|------|--------|------|
| `OA_USER_NAME` | ✅ | - | OA用户名 |
| `OA_USER_PASSWD` | ✅ | - | OA密码 |
| `OA_STATE_FILE` | ❌ | `~/.oa-todo/login_state.json` | 登录状态文件 |
| `OA_DB_PATH` | ❌ | `~/.oa-todo/oa_todos.db` | 数据库路径 |
| `OA_TODOS_DIR` | ❌ | `~/.oa-todo` | 待办目录 |
| `OA_DETAILS_DIR` | ❌ | `~/.oa-todo/details` | 详情目录 |
| `LOGIN_TIMEOUT_MINUTES` | ❌ | 25 | 登录状态超时（分钟） |
| `PAUSE_TIMEOUT_MINUTES` | ❌ | 10 | 断点超时时间（分钟） |
